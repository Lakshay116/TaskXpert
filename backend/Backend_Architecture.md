# Jira + Chat Support System - Backend Architecture Documentation

## 1. System Overview
The backend is a robust RESTful API built with **Node.js** and **Express.js**, utilizing **PostgreSQL** for relational data storage. It features a real-time WebSocket server using **Socket.io** for live chat messaging.

### Tech Stack
- **Runtime:** Node.js
- **Framework:** Express.js
- **Database:** PostgreSQL (Hosted on Aiven)
- **Authentication:** JSON Web Tokens (JWT) & Google OAuth
- **File Storage:** Cloudinary
- **Real-Time:** Socket.io
- **Security:** Helmet, Express-Rate-Limit, bcrypt, CORS

---

## 2. Security & Middleware
- **Helmet:** Sets secure HTTP headers.
- **CORS:** Allows cross-origin requests from the frontend.
- **Rate Limiting:** Limits requests (100 requests per 15 minutes per IP) to prevent brute force attacks.
- **JWT Protection:** `protect` middleware ensures only valid users access private routes.
- **RBAC (Role-Based Access Control):** `authorize` middleware restricts endpoints to specific roles (`Admin`, `Manager`, `Employee`, `Agent`).

---

## 3. Database Schema Overview
- **Roles:** Stores role definitions (`Admin`, `Manager`, `Employee`, `Agent`).
- **Users:** Stores user profiles, hashed passwords, and `google_id`.
- **Projects:** Stores project details and ownership.
- **Tasks:** Linked to projects. Assigned to users. Tracks status, priority, and due dates.
- **Comments:** Linked to tasks and users.
- **Attachments:** Stores Cloudinary URLs linked to tasks.
- **Tickets:** Support requests generated by users, assignable to Agents.
- **Ticket Messages:** Chat history for tickets.

## 4. API Endpoints Reference

### 🔐 Authentication Module (`/api/auth`)
| Method | Endpoint | Description | Access |
|---|---|---|---|
| POST | `/register` | Register a new user | Public |
| POST | `/login` | Authenticate and get JWT | Public |
| POST | `/google` | Google OAuth login/signup | Public |
| POST | `/refresh-token` | Get a new access token | Public |

### 📁 Project Module (`/api/projects`)
| Method | Endpoint | Description | Access |
|---|---|---|---|
| POST | `/` | Create a new project | Admin, Manager |
| GET | `/` | Get all projects | Authenticated |
| GET | `/:id` | Get specific project | Authenticated |
| PUT | `/:id` | Update project details | Admin, Manager |
| DELETE | `/:id` | Delete a project | Admin, Manager |

### 📋 Task Module (`/api/tasks`)
| Method | Endpoint | Description | Access |
|---|---|---|---|
| POST | `/` | Create a new task | Admin, Manager, Employee |
| GET | `/project/:projectId`| Get all tasks for a project | Authenticated |
| GET | `/:id` | Get task details | Authenticated |
| PUT | `/:id` | Update a task | Authenticated |
| DELETE | `/:id` | Delete a task | Admin, Manager, Employee |

### 💬 Comment Module (`/api/comments`)
| Method | Endpoint | Description | Access |
|---|---|---|---|
| POST | `/` | Add a comment to a task | Authenticated |
| GET | `/task/:taskId` | Get comments for a task | Authenticated |
| DELETE | `/:id` | Delete a comment | Owner, Admin, Manager|

### 📎 Attachment Module (`/api/attachments`)
| Method | Endpoint | Description | Access |
|---|---|---|---|
| POST | `/` | Upload a file to a task | Authenticated (FormData) |
| GET | `/task/:taskId`| Get attachments for a task | Authenticated |
| DELETE | `/:id` | Delete an attachment | Authenticated |

### 🎟️ Ticket Module (`/api/tickets`)
| Method | Endpoint | Description | Access |
|---|---|---|---|
| POST | `/` | Create a support ticket | Authenticated |
| GET | `/` | Get tickets (User's or All) | Authenticated |
| GET | `/:id` | Get specific ticket | Authenticated |
| PUT | `/:id` | Update status/assignee | Admin, Manager, Agent |
| DELETE | `/:id` | Delete ticket | Admin, Manager |

### ✉️ Real-Time Chat (Socket.io & HTTP)
**HTTP Endpoints (`/api/messages`)**
| Method | Endpoint | Description | Access |
|---|---|---|---|
| GET | `/ticket/:ticketId`| Get historical chat messages | Ticket Owner, Admins, Agents |

**WebSocket Events (Socket.io)**
- `join_ticket`: Connects the user to a specific ticket room.
- `send_message`: Emits a new message, saves it to PostgreSQL, and broadcasts it to the room.
- `receive_message`: Listens for incoming messages from the room.
